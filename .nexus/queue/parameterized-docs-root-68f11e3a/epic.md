# Epic: Parameterized Docs Root

## Description

Where a repo keeps its human-readable docs is a per-repo fact, but the toolchain treats it as a
constant: the atlas generator defaults its output to `docs/concepts.md` and bakes a
`../.nexus/concepts/` link prefix into the rendered atlas, `/nxs.distill` names that path in its
regenerate step, sync gate, and staged file set, and the cross-ref skill unconditionally strips a
leading `docs/` when building absolute URLs. That constant is wrong for a docs-only hub, whose
human docs live at its repo root. Today such a hub must carry a `docs/` folder that exists only to
satisfy the tooling, and any `docs/`-rooted path routed through the cross-ref skill resolves to a
dead link.

This epic makes the docs root a resolved property of workspace context. An explicit configuration
wins when present; otherwise the role decides the default — `docs/` for a single-repo project and
for multi-repo members (unchanged from today), the repo root for a hub. The workspace resolver is
the sole producer of the value, extending the same rule that already pins the hub's
portable-tooling location in resolved context (#44); consumers read it instead of re-deriving it.
The derived-artifact and link surfaces then honor it: the atlas lands at `<docs-root>/concepts.md`
with its internal links computed from its actual location, the hub drain regenerates and stages
that path, and cross-ref URL building strips the resolved docs root rather than a hardcoded
`docs/`.

Single-repo projects see no change: their resolved docs root is `docs/`, the atlas stays at
`docs/concepts.md`, and generated bytes are identical. This amends the "generated atlas at
`docs/concepts.md`" contract item of decision `0b8973e2` — the path becomes
`<docs-root>/concepts.md`, whose single-repo value is unchanged.

## Success Metrics

- A docs-only hub with human docs at its root completes a drain with the atlas at the hub root,
  every atlas link resolving, and no `docs/` directory created anywhere in the run.
- A single-repo project produces a byte-identical atlas at the unchanged `docs/concepts.md`
  location before and after this epic.
- The cross-ref skill produces resolvable URLs for docs paths in both layouts — zero dead links
  from the strip rule.
- The resolved docs root is visible in the workspace status read-out for every repo it reports.

## User Stories

### Story 1: Docs root joins resolved workspace context

- **story_type:** system
- **size:** M

**As a** toolchain consumer (command or portable tool), **I want** the workspace resolver to
produce the docs root for every repo it describes, **so that** one authoritative producer replaces
scattered hardcoded `docs/` assumptions.

#### Acceptance Criteria

- [ ] **Given** a checkout carrying a hub manifest and no explicit docs-root configuration,
      **when** the resolver runs, **then** the resolved context reports the hub's docs root as the
      repo root.
- [ ] **Given** a checkout carrying a member pointer, or carrying neither workspace artifact
      (single-repo mode), **when** the resolver runs, **then** the resolved docs root is `docs/`.
- [ ] **Given** an explicit docs-root configuration, **when** the resolver runs, **then** the
      explicit value is reported and the role-based default is ignored.
- [ ] **Given** the same workspace resolved from the hub checkout and from a member checkout,
      **when** both resolutions complete, **then** the reported docs roots are identical for every
      repo (the existing deep-equal parity guarantee extends to the new field).
- [ ] **Given** any checkout, **when** the workspace status read-out prints, **then** it includes
      the resolved docs root.

#### Notes

The configuration surface for the explicit override (hub manifest vs. settings) is a design
decision for the decision record; the story requires only that the resolver is the single producer
and no consumer re-derives the value (workspace-resolution invariant 3).

### Story 2: Atlas generator derives its location and links from the docs root

- **story_type:** system
- **size:** M

**As a** hub maintainer, **I want** the atlas generator to write to the resolved docs root and
compute its links from where the atlas actually lands, **so that** the atlas is correct in any
repo layout instead of only one directory below the repo root.

#### Acceptance Criteria

- [ ] **Given** a repo whose resolved docs root is `docs/`, **when** the generator runs with no
      `--out`, **then** the atlas is written to `docs/concepts.md` with content byte-identical to
      the pre-change output.
- [ ] **Given** a hub whose resolved docs root is the repo root, **when** the generator runs with
      no `--out`, **then** the atlas is written to `concepts.md` at the repo root, no `docs/`
      directory is created, and every concept link in it resolves to its
      `.nexus/concepts/<slug>.md` page from the atlas's location.
- [ ] **Given** any explicit `--out` path, **when** the generator runs, **then** the atlas's
      internal links are computed relative to that location and resolve.
- [ ] **Given** `--check` mode, **when** it runs, **then** it compares against the same resolved
      location write mode uses, in both layouts.
- [ ] **Given** the parity corpus, **when** the in-repo tooling and a fresh portable build both
      run, **then** atlas outputs are byte-identical (portable-tooling invariant 3 holds under the
      new path logic).

#### Notes

The portable build may consult the resolver at run time; `derive-entry-diff` set that precedent
(#54). Today's rendered link prefix `../.nexus/concepts/` is the hardcoding this story removes.

### Story 3: Cross-ref URL building strips the configured docs root

- **story_type:** system
- **size:** S

**As a** command author placing durable doc links in issue bodies, **I want** absolute-URL
conversion to strip the resolved docs root instead of a hardcoded `docs/` prefix, **so that**
links resolve in repos whose docs live at the root.

#### Acceptance Criteria

- [ ] **Given** a resolved docs root of `docs/` and a cross-ref URL ending in `/docs`, **when** a
      `docs/`-prefixed path is converted, **then** the output URL is unchanged from the pre-change
      behavior.
- [ ] **Given** a resolved docs root of the repo root and a cross-ref URL pointing at the repo
      root, **when** any repo-relative docs path (e.g. `concepts.md`) is converted, **then** the
      URL is `<cross-ref-root>/<path>` with nothing stripped, and fetching it does not 404 against
      the repo's actual layout.
- [ ] **Given** any resolved docs root (explicit or role-default), **when** a path carrying that
      root prefix is converted, **then** exactly the resolved prefix is stripped once; no
      unconditional `docs/` strip remains in the conversion.
- [ ] **Given** a resolved docs root and a cross-ref URL whose trailing segment does not match
      that root (e.g. a repo-root docs root but a URL still ending in `/docs`), **when** a path
      is converted, **then** the skill surfaces the mismatch as operator error rather than
      emitting a link that would dead-end against the repo's actual layout.

### Story 4: The hub drain follows the resolved atlas location

- **story_type:** system
- **size:** S

**As a** hub maintainer running `/nxs.distill`, **I want** the drain's regenerate step, sync gate,
and staged paths to follow the resolved docs root, **so that** a drain never recreates a `docs/`
folder the repo doesn't use.

#### Acceptance Criteria

- [ ] **Given** a hub with docs at its root, **when** a drain completes, **then** the regenerated
      atlas is at the hub root, the atlas sync gate passes against that path, the drain's staged
      file set includes it, and no `docs/` directory exists afterward.
- [ ] **Given** a single-repo project, **when** a drain completes, **then** behavior is unchanged:
      the atlas is regenerated and staged at `docs/concepts.md`.
- [ ] **Given** the drain's completion report, **when** it names the atlas, **then** it reports
      the resolved path rather than a fixed literal.

## Assumptions

- Hub-root as the hub default reflects the docs-only-hub convention; a hub that keeps human docs
  under `docs/` sets the explicit override rather than the default flipping per-repo.
- Multi-repo members keep `docs/` as their default — the hub role, not multi-repo membership, is
  what moves the default.
- The vendored portable-tools bundle is re-vendored (with its fingerprint pin) as part of
  delivering the generator change; that is the existing release mechanism, not new scope.

## Out of Scope

- Planning-surface paths following the docs root — `/nxs.epic`, `/nxs.close`, and `/nxs.setup`
  still write `docs/features/…`, `docs/product/…`, and `docs/system/…` literals. A hub planning
  its own docs epics will still create `docs/` through those commands; routing them through the
  resolved docs root is a follow-on epic.
- Migrating existing hubs (e.g. relocating an already-committed `docs/concepts.md` to the hub
  root) — a consumer-side chore once this ships.
- Any change to the meaning of the `cross-ref.docs-root` URL setting; it remains the URL of the
  docs root. Only the local strip rule keyed to it changes.
