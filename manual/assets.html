Explainer

Nexus, explained by its assets

Nexus is a lean, spec-driven delivery pipeline for the age of AI agents. It turns intent into validated, decision-grade specs — and leaves implementation to engineers. The pipeline is easiest to understand not by its commands, but by the small set of files it produces and consumes. This deck walks through each one: what it is, and why it earns its place.

Why these assets, and no others

Generation is cheap. Judgment is not.

AI can produce planning artifacts endlessly. The failure mode Nexus guards against is speculative over-generation: sprawling design docs, per-task plans, and prose reports produced ahead of validated scope — burying the few decisions a human actually needs to make.

Every artifact must force a human decision, or it gets cut.

That razor shaped the whole asset set. There is no 16-section design document, no per-task plan, no status report. Each surviving asset exists because it either forces a judgment (approve this scope, resolve this question, accept this deviation) or preserves a judgment so it never has to be re-made.

Consequence: assets are small, few, and each has exactly one job.

The map

Three surfaces, split by consumer

Every asset lives on exactly one surface — and placement is decided by who consumes it, never by content type or by who wrote it. The split is physical, so a misplaced artifact is a visible review violation, not a judgment call.

One epic’s journey

How the assets hand off to each other

Intent becomes an epic.md

A capability description is shaped into a right-sized epic with user stories and acceptance criteria. On human approval, the epic and its story issues are filed on GitHub together.

Design judgment lands in decision-record.md

Key decisions, refuted alternatives, invariants, and blocking risks — the focused “why” beside the epic in its queue folder.

Engineers implement the stories

Nexus plans and gates the work; it does not write the code. In-flight decisions optionally land in gitignored scratch.

Conformance is checked — analyze-receipt.md

The build is compared against acceptance criteria, success metrics, and the decision record’s invariants. The receipt gates the close.

The epic closes with a close-record.md

Decisions made during implementation, plus rationale for each deviation the diff revealed. Deferred scope goes to the feature backlog; the process lesson gets its own file.

After merge, the queue entry is distilled

A reviewed distillation-PR drains the folder into concept pages and refreshed code anchors, regenerates the atlas, then deletes the consumed entry. The judgment survives; the scaffolding doesn’t.

Committed queue · container

The committed queue

.nexus/queue/<epic-slug>-<id>/

What it is

One committed folder per epic, holding that epic’s human planning artifacts: the epic, the decision record, the analyze receipt, the close record. It is the single handoff surface between the delivery pipeline and the knowledge store.

It also accepts decision-only memos — a single reviewed file recording an out-of-band decision with no code diff.

Why it’s useful

  • Committed, not gitignored — rationale is visible from every checkout and travels to main with the PR. Durability is structural, not procedural.
  • Presence is the only state. An unconsumed entry is one that still exists — no status files to read or forget.
  • Abandoned epics never distill. They never reach the trunk, so bad plans can’t contaminate the knowledge store.
  • Everything in it passed a human gate. Ungated captures (e.g. plan-mode plans) never enter.
Filled by
each pipeline stage in turn
Drained by
the distiller, after merge
Lifespan
epic lifetime; deleted on drain, recoverable via git history
Committed queue · planning

The epic

.nexus/queue/<epic>/epic.md

What it is

A right-sized unit of capability: a short description, measurable success metrics, and a handful of user stories with Given/When/Then acceptance criteria, sizes, an implementation sequence with dependencies — plus explicit assumptions and out-of-scope lines.

The story is the terminal planning unit: decomposition stops once a story is small enough to ship and verify on its own. Each story becomes one GitHub issue, filed as a sub-issue of the epic — there is no task layer beneath it.

Why it’s useful

  • It forces the scope decision. Approval happens at a single decision-grade digest; only then are the epic and story issues filed, together.
  • Acceptance criteria are the contract — the analyze stage later checks the build against exactly these lines.
  • Oversized scope can’t sneak through: it decomposes into backlog stubs instead of bloated epics.
  • Complexity is declared (S/M/L/XL with named drivers), which tiers how much design rigor the next stage must bring.
Produced by
/nxs.epic, from natural-language intent
Consumed by
engineers, the analyze gate, the distiller
Forces
one human approval of scope + stories
Committed queue · design

The decision record

.nexus/queue/<epic>/decision-record.md

What it is

The focused architectural “why” — it replaced a 16-section HLD. Key decisions each record what was decided, why, and the refuted viable alternative (never a strawman). Alongside: hard constraints and invariants the build must preserve, risks that block progress, and open clarifications only a human can resolve.

Sections are tiered by complexity: small epics need only Key Decisions and Invariants; large ones must fill everything.

Why it’s useful

  • No file paths, type names, or API specs — those rot against source and belong to the engineer. What remains is exactly the part that can’t be regenerated from code.
  • Refuted alternatives prevent re-litigation. Six months later, nobody re-argues a decision whose losing option and trade-off are written down.
  • Invariants become checkable: the analyze stage verifies the build preserved them.
  • It is the distiller’s primary rationale source — this prose becomes concept-page decision-log entries.
Produced by
/nxs.hld, reading the queued epic
Consumed by
engineers, the analyze gate, the distiller
Forces
resolution of open clarifications before build
Committed queue · conformance

The analyze receipt

.nexus/queue/<epic>/analyze-receipt.md

What it is

The small proof that a conformance pass ran: after implementation, the code is checked against the epic’s acceptance criteria, its success metrics, and the decision record’s invariants — does the build do what the planning said? Findings are reported inline; the receipt records that the check happened, when, and at what commit.

Why it’s useful

  • It gates the close. An epic cannot close without a present, current receipt — or an explicit, recorded human waiver.
  • Staleness is detectable: a receipt tied to an old commit doesn’t count, so the check can’t silently drift from what ships.
  • It keeps conformance honest without producing a prose report nobody reads — the receipt is evidence, not literature.
Produced by
/nxs.analyze, after stories are implemented
Consumed by
the close gate
Forces
fix, waive, or don’t close
Committed queue · closure

The close record

.nexus/queue/<epic>/close-record.md

What it is

Human prose written at epic close: decisions made or changed during implementation, and — via a close-from-diff pass — rationale for each place the shipped diff deviates from the decision record. Matched work needs no entry.

It carries pointers, not payloads: deferred scope is appended to the feature backlog; the process lesson becomes its own file. The record just points at both.

Why it’s useful

  • The diff does the remembering. The human isn’t asked to “write a summary” — only to explain detected deviations. Targeted questions get honest answers.
  • It captures the why that only exists at close: what changed mid-build and what debt was accepted — the distiller cannot derive this from the code diff alone.
  • Nothing is duplicated: scope lives where it will be re-triaged, lessons live where they’ll be read.
Produced by
/nxs.close (also closes the GitHub issues)
Consumed by
the distiller, as its close-time “why” source
Precondition
all story issues closed + current analyze receipt
Human surface · ground truth

Product context

docs/product/context.md

What it is

The durable product ground truth, built once through a short strategic interview: vision, explicit anti-goals, product principles, personas with jobs-to-be-done and pain points, competitive landscape, a north-star metric, and impact/effort thresholds for scoring.

Why it’s useful

  • Every epic inherits it. Personas and principles don’t get re-invented per epic — stories reference this file instead of restating them.
  • Anti-goals kill scope creep early: a written “does not do X” settles arguments before they start.
  • Impact thresholds make prioritization arguable in numbers, not adjectives.
  • Inferences from the interview are marked as inferences — the file is honest about what’s confirmed vs. assumed.
Produced by
the setup interview (≤5 questions + PM inference)
Consumed by
epic generation, design, anyone new to the product
Lifespan
durable; refreshed when strategy shifts
Human surface · scope ledger

Feature homes & backlogs

docs/features/<feature>/README.md · backlog.md

What they are

Each feature gets a tiny README — one paragraph plus links to its epics’ GitHub issues — and an append-only backlog: a re-triage queue of stubs, each with a goal, an estimate, dependencies, provenance, and candidate stories.

Two writers feed the backlog: epic planning (when oversized scope is decomposed into stubs) and epic close (deferred scope). One consumer reads it: the next epic, which promotes a stub into a full epic.

Why they’re useful

  • Cut scope has somewhere honest to go. “Not now” doesn’t mean “lost” — it means a stub with provenance, waiting for re-triage.
  • Stubs are cheap by design: a goal and candidate stories, not a speculative full plan for work that may never be chosen.
  • blocked_by lines make sequencing visible before anyone commits to an epic.
  • Status is auditable: proposed → promoted, append-only, with dates.
Written by
epic decomposition & close-time deferral
Consumed by
the next epic planning pass
Forces
a deliberate promote/defer decision per stub
Human surface · process memory

Delivery lessons

docs/delivery/lessons/<date>-<slug>.md

What they are

One short file per lesson, captured at epic close: what happened, what we learned, what to change next time, with the source epic in frontmatter. These are process learnings — how the work went — deliberately separate from the system knowledge that flows into concept pages.

Real examples from this repo: “verify mockup assumptions with a diff”, “isolate the hard concern first”, “front-load the Nx target risk.”

Why they’re useful

  • One lesson per file keeps each one findable and quotable — no scrolling a mega-retro document.
  • The close stage extracts it while memory is fresh, instead of relying on a quarterly retro reconstructing events.
  • Short and actionable by rule: a lesson nobody reads costs context for nothing.
Produced by
epic close (or by hand, same format)
Consumed by
the team, before similar work
Lifespan
permanent
Human surface · derived

The concept atlas

docs/concepts.md

What it is

A generated orientation map of the machine-side concept store: one line per concept, grouped by area, each linking to its full page. It is machine-written but human-consumed — which, under the consumer rule, is exactly why it lives on the human surface, with its derived ownership declared in a header comment.

Why it’s useful

  • It’s the human on-ramp to the concept store — you skim one page instead of listing a directory of slugs.
  • Never hand-edited, never stale by accident: regenerated on every distillation drain, with a checker that verifies atlas ↔ store sync.
  • It demonstrates the placement rule: authorship never decides where an asset lives; the consumer does.
Produced by
generator script, on every drain
Consumed by
humans orienting in the knowledge store
Hand-edits
never — derived, rebuilt not fixed
Machine surface · knowledge

Concept pages

.nexus/concepts/<slug>.md

What they are

One distilled concept per file — current behavior, hard invariants, integration points, and an append-only decision log where every entry carries its provenance (the issue it came from), the why, and the refuted alternative. Bodies are capped at ~400 words; code, file paths, and type names are banned so pages describe behavior, not implementation.

Retrieval is grep-native: no graph engine, no embeddings — just readable markdown searched by slug, alias, or phrase. The touches: field answers “what’s the blast radius of changing X?” with a one-line search.

Why they’re useful

  • They hold the judgment code can’t express — why the system is this way — so they’re git-tracked and reviewed like source, not regenerated.
  • Future planning reads them instead of re-deriving system state, so decision records stay small.
  • The append-only log means history is never rewritten — a reversed decision gets a new entry, keeping the reasoning chain intact.
  • Single producer: pages change only through a reviewed distillation-PR — the store can’t drift via casual edits.

Code anchors

.nexus/anchors/<slug>.md

What they are

A derived sidecar per concept: the list of source files that implement it, each with a one-line role — SHA-stamped with the commit they were generated from. Regenerated by the distiller on every drain that touches the concept; never hand-edited (stale anchors are rebuilt, not fixed).

Why they’re useful

  • They resolve a real tension: concept pages ban file paths because paths rot — but an agent still needs the concept → code hop. Anchors carry the rot-prone half in a disposable, regenerable file.
  • The SHA stamp makes staleness detectable instead of silent: you know exactly which commit the map reflects.
  • Judgment and mechanics age differently — splitting the durable why (page) from the perishable where (anchor) lets each be maintained on its own schedule.

Committed scratch · ungated

Scratch capture

.nexus/queue/<epic>/<username>/decisions-<branch>.md  ·  committed

What it is

A committed, per-user, per-branch scratch file inside the epic’s queue entry for in-flight decision stubs — written at the moment a non-obvious implementation choice is made: the choice, one sentence of why, and the refuted alternative. Nothing more. Working notes sit beside it as notes-<branch>.md.

Because it is committed, it rides the PR to review and to the lead’s checkout at merge. The close pass mines it as hints, verified against the diff; the distiller then drains it with the entry when the distillation-PR merges. Scratch content never enters the concept store directly — only close-gated prose crosses over.

Why it’s useful

  • It saves what would otherwise evaporate: mid-build rationale is far higher-fidelity in the moment than reconstructed from memory weeks later at close.
  • Ungated by design, so it’s worthless as record but cheap as capture — the gate is applied later, by a human, at close.
  • Branch-keyed filenames make it conflict-free: each session appends to its own file, so any number of engineers or branches touch the same epic without a merge conflict.
